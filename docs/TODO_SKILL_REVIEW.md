# Skill Review System - Future Improvements

## TODO

### 1. 隔离环境/沙盒测试 (Priority: High)

**问题**: 当前 AI 审核仅静态分析 SKILL.md 文件，无法验证技能实际运行效果，也无法检测运行时安全问题（如命令注入、恶意代码执行等）。

**目标**: 建立隔离执行环境，在沙盒中测试技能

**方案选项**:
- [ ] 使用 Docker 容器作为沙盒，限制 CPU/内存/网络
- [ ] 使用 Firecracker MicroVM 提供更轻量的隔离
- [ ] 使用 gVisor 等用户态内核实现更强的安全隔离
- [ ] 预置常见测试场景，验证技能功能正确性
- [ ] 记录技能执行过程中的系统调用、文件访问、网络请求

**安全检查项**:
- 敏感文件访问（~/.ssh, /etc/passwd 等）
- 可疑网络连接
- 命令注入检测
- 资源消耗限制（防止 DoS）

---

### 2. 更健壮的 AI 审查 (Priority: Medium)

**问题**: 当前自动审核仅基于简单的正则匹配，缺乏语义理解和深度分析能力。

**目标**: 构建专业的 AI 审核 Agent，提供更全面、准确的审查

**改进方向**:
- [ ] 接入 LLM 进行深度语义分析（当前是基础正则检查）
- [ ] 建立技能知识库，对比相似技能检测重复/抄袭
- [ ] 静态代码分析（检查脚本语言的常见漏洞模式）
- [ ] 依赖安全检查（扫描已知 CVE 的依赖版本）
- [ ] 多轮审查机制（初审 -> 深度分析 -> 综合评估）
- [ ] 学习能力（根据管理员反馈优化审查标准）

**审查维度扩展**:
| 维度 | 当前 | 目标 |
|-----|-----|-----|
| 格式检查 | 基础 frontmatter | 完整 schema 验证 |
| 安全扫描 | 无 | 静态 + 动态扫描 |
| 功能测试 | 无 | 沙盒执行验证 |
| 代码质量 | 无 | 风格、复杂度分析 |
| 文档完整度 | 基础检查 | 示例、参数说明 |

### 3. 聊天对话中提交技能 (Priority: Low)

**问题**: 当前用户需要到技能商店页面手动提交技能，不够便捷。

**目标**: 用户可以在聊天对话中通过自然语言指令提交技能

**场景示例**:
```
用户: 把这个技能提交到精选商店
Agent: 已为您提交技能 "web-search" 到审核队列，请等待审核结果。
```

**实现方案**:
- [ ] 扩展 `skill-review` skill，支持 `submit_to_curated` 指令
- [ ] 用户说"提交这个技能到精选"时，Agent 调用提交 API
- [ ] 支持确认流程（显示技能信息，用户确认后提交）
- [ ] 提交后主动告知用户审核状态查询方式

### 4. 技能版本升级审核 (Priority: Low)

**问题**: 当前技能更新需要重新走完整审核流程，效率低。

**目标**: 支持 diff 模式审核，只审查改动部分

**实现方案**:
- [ ] 记录技能版本历史
- [ ] 对比新旧版本生成 diff
- [ ] AI 审核 focus 在改动部分
- [ ] 小改动（如文档更新）可快速通过

- [ ] 已上架技能的更新版本优先于新技能
- [ ] 长时间未审核的任务自动提升优先级

### 8. 技能组合包 (Skill Bundles) (Priority: Medium)

**背景**: 之前讨论过 "禁用技能" 的设计，结论是技能组合包比简单的禁用更合理。

**问题**: 当前每个 Agent 可以启用/禁用单个技能，但缺乏预设的技能组合管理。

**目标**: 支持创建和管理技能组合包

**方案**:
- [ ] 预定义技能组合包（如 "开发工具包"、"DevOps 套件"）
- [ ] 用户自定义组合包
- [ ] 创建 Agent 时选择安装哪些组合包
- [ ] 组合包内技能可单独启用/禁用

**优势**:
- 比单个禁用更清晰
- 便于批量管理技能
- 可共享组合包配置

### 9. 技能商店独立服务评估 (Priority: Low)

**背景**: 之前讨论过精选技能商店是否有必要作为单独服务。

**当前架构**: 技能商店是 Platform Gateway 的一部分
- `CuratedSkill` 模型管理元数据
- 文件存储在本地文件系统
- 审核流程内嵌在 skills.py 路由中

**考虑独立服务的情况**:
- [ ] 技能数量激增（>10,000）
- [ ] 需要支持第三方开发者入驻
- [ ] 需要独立的发布/回滚流程
- [ ] 需要版本管理、依赖解析等复杂功能

**暂不独立的原因**:
- 当前规模较小，单体架构足够
- 减少运维复杂度
- 共享用户认证和数据库

**未来触发条件**: 当技能数量和用户量达到一定程度后再评估

**背景**: 之前讨论过审核通过后，除了站内通知，是否要让审核 agent 主动发消息给用户的主 agent。

**当前实现**: 仅站内通知（右上角铃铛）

**可选增强**:
- [ ] 审核完成后，skill-reviewer agent 主动发消息给用户主 agent
- [ ] 用户打开聊天界面就能看到审核结果

**暂不实现原因**:

- 避免过多消息打扰用户
- 可先观察用户使用情况再决定是否添加
